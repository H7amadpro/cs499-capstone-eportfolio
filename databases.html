Artifact Description

The artifact being referenced is the same application created in the last two enhancements ‚Äî the iOS inventory management application. The original inventory management application was created in CS 360: Mobile Architecture and Programming with a basic database (SQLite) to track items with an item name, quantity, and timestamp.

The original database used a flat schema with no sophisticated relationship definitions or integrity checks between the various tables and there were no audit capabilities or consideration of backup mechanisms to protect the data, which was acceptable in the CS 360 course project, but the requirements of a fully-functioning production inventory management system are much different than a CS 360 course project.

Justification for Inclusion

I selected this artifact as it provides the best opportunity to demonstrate the skills I have gained in the development and enhancement of databases. Advanced database models are required when designing a modern inventory system.

The following areas provide evidence of my skills:

Comprehensive SwiftData Schema Design

I have created a complete database schema containing eight interconnected entities: Item, CategoryEntity, User, Branch, Supplier, AuditLogEntry, ItemImage, and StockTransfer. All of these entities were carefully designed, with appropriate property definitions and relationships.

Entity Relationships with Referential Integrity

I designed all entities with referential integrity using the @Relationship property wrapper in SwiftData. This creates valid relationships between the entities so the data contained in those relationships can be used correctly and efficiently in queries. Additionally, delete rules were created for each relationship type.

Complete CRUD Functionality

I designed service classes for each entity to support all CRUD operations. The service class for each entity contains error handling, data validation, and audit logging capabilities consistent with the design and standards of the industry.

Comprehensive Audit Logging

AuditLogEntry entity and AuditService class functionality were designed to provide a complete auditing system. All significant activities conducted on the application are recorded in an audit log with a timestamp, associated user, and before/after values. This provides a method for ensuring compliance and the ability to monitor the security of the application.

Backup and Restore Functionality

I have created a BackupService class for users of the application to back up and restore their data. The BackupService functionality allows users to back up their data, and in the event of data loss, users may restore their data.

Multiple Location Inventory Management

The database design can support multiple locations of physical inventory, and allow for stock transfers between locations via the use of the StockTransfer entity and stock transfer status, and approval tracking.

Reflection on Enhancement Process

The most challenging yet rewarding component of designing and developing my database was enhancing the relationships within the database. I learned to view data in terms of how each piece of data relates to one another, the integrity of that data, and how to support the ongoing maintenance of the database for its entire lifespan.

Over-Normalization of Data

One of the largest challenges I faced was determining how much normalizing needed to take place in developing a schema while developing an efficient way to retrieve data. I had over-normalized my database, which resulted in my database producing complex queries and had performance issues, but eventually I was able to find the balance between normalizing and denormalizing my schema.

Audit Logging Performance

The second challenge I faced was implementing an audit logging functionality into my application without negatively impacting the performance of the application. I was able to accomplish this through implementing the Auditing system using asynchronous logging and batching audit log entries into batches where applicable.

Incorporating Feedback

I was able to improve my database by adding indexes on fields frequently queried due to feedback from code reviews, and creating the ability to soft-delete inventory item records, providing a way for users to recover deleted item records.
